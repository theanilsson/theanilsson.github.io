Battle Royale Simulation Using Heatmaps

(The simulations running below are GIF images, keybinds will not work)
  • View the implementation on GitHub

Summary

Heatmaps are the core of my simulator's systems. A Heatmap is usually implemented using a graph data structure of some kind, spanning the area of the game world you want to track activity in, where each graph node tracks activity at that point of the graph as "heat" on their part of the "map". In my implementation, each Heatmap consists of a 2D grid of cells (üîç) that track a floating-point heat value between 0 and 1. Cell size is adaptable for each Heatmap, allowing some types of data to be collected with more precision (e.g. player positions) and others over a larger area where precision is less important (e.g. strong weather).

I intentionally made the game rounds run much faster than in a typical Battle Royale game to allow for many more simulations while still producing results representative of a longer round. This not only provides a larger dataset to work with but also enables faster testing and iteration on individual simulation components. Additionally, I kept the game objects intentionally simple so I could focus most of my time on developing the Heatmaps and algorithms. The air drops, loot piles, and AI pawns only needed to emit valuable data onto the Heatmaps so that the surrounding systems could function properly.

Though it would be fun to create AI that behaves like real players in a Battle Royale game (maybe I will in the future), that wasn't the goal of this project. Instead, it serves as a showcase of how data can be continuously collected and utilized over the course of such a game, as well as how that process can be implemented effectively.

Avoiding Predictability

A common problem in competitive games is predictability and balancing. I wanted to try to develop a kind of solution to that problem using game events that adapt to the state of the game round and each player's actions.

Start of simulation object spawns avoid previous round to promote more variation in subsequent simulation rounds and thus more diverse results.

Avoiding Areas With High Death Tolls

The safe zone avoids areas with high death tolls to try to prevent players clustering up too early in the round.

The Anti Camping Mechanic

The safe zone avoids areas where players have spent a lot of time to try to force them to stay on the move.

Strategic Positioning Balancing

The safe zone avoids areas where a lot of loot has been picked up to avoid giving an advantage in strategic positioning to players who already have an advantage from gear picked up.

The Procedurally Generated Weather System

The safe zone is slightly affected by the winds (simulated by high heat values) of the weather system, shifting it's shrinking direction ever so slightly towards the area with the least windy conditions.

The Air Drops

To disincentivize camping and prevent players who looted previous air drops in the current round from gaining too much of an advantage, new air drops spawn in the coldest cluster within the current safe zone. This usually means the drop is positioned as far as possible from previous air drop locations without being placed at the very edge of the safe zone. Depending on the pattern of previous air drops, later drops may sometimes land near earlier ones, but never in the exact same spot.
